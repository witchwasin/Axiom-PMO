# Axiom-PMO Community

Axiom-PMO is for teams that want AI-assisted delivery without giving up
traceability, human authority, or release discipline.

This community exists to improve the practical controls that keep AI agents
inside verified requirements, approved scope, evidence-backed delivery, and
human-approved release gates.

## What We Work On

Community contributions are most useful when they make Axiom-PMO clearer,
safer, easier to adopt, or easier to verify.

Good contribution areas include:

- validation rules, rule catalog updates, and deterministic test fixtures;
- project templates for Lite, Standard, and Strict operating modes;
- examples that show realistic source-backed delivery without unnecessary
  documentation weight;
- documentation that helps teams adopt the framework safely;
- integrations with AI coding tools and delivery workflows;
- governance patterns for approvals, evidence, QA, release readiness, and
  human-controlled git operations.

## How We Make Decisions

Axiom-PMO favors controls that are explicit, reviewable, and testable.

When proposing a change, explain:

- the problem it solves;
- the source, workflow, or risk that justifies it;
- whether it affects validation strictness, approval gates, evidence
  requirements, or human authority;
- what positive and negative cases prove the behavior.

Changes that weaken governance must be treated as policy decisions, not quick
fixes. A rule should not become less strict simply because a test is difficult
to satisfy.

## Contribution Principles

- Evidence over assertion.
- Deterministic checks over persuasive prompts.
- Human approval over agent self-approval.
- The smallest process that controls the real risk.
- Source files are user-owned and should not be edited by automation.
- Private data, customer material, secrets, and raw PII do not belong in issues,
  examples, fixtures, or pull requests.

## Good First Contributions

Useful starter work includes:

- clarifying confusing documentation;
- improving examples without adding new process weight;
- adding missing negative fixtures for validator behavior;
- proposing rule descriptions that make failures easier to understand;
- documenting an integration pattern for a real AI coding workflow.

If a proposed change touches permissions, approvals, release authority,
sensitive data handling, or external integrations, treat it as high-risk and
describe the governance impact clearly.

## Community Standards

Be precise, practical, and respectful. Challenge weak controls, unclear
evidence, and unsafe automation behavior, but do it in a way that helps the
project improve.
