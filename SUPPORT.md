# Support

Thank you for using Axiom-PMO. This project is a governance control plane for
AI-assisted delivery, so clear reports and safe examples matter.

## Where To Ask

Use GitHub issues for:

- validator or doctor defects;
- documentation gaps;
- template improvements;
- validation rule proposals;
- integration requests;
- reproducible problems with examples or fixtures.

Before opening an issue, please check the README, CONTRIBUTING guide, and
existing issues when possible.

## What To Include

For bug reports, include:

- Axiom-PMO version from the `VERSION` file;
- operating system and PowerShell version;
- the command you ran;
- the relevant validation output;
- a minimal project or fixture that reproduces the problem.

For rule or policy proposals, include:

- the risk or workflow gap being addressed;
- the proposed rule behavior;
- expected severity;
- positive and negative examples;
- whether the change affects approval gates, evidence requirements, release
  readiness, or git authority.

## Private Data

Do not include private data in public issues, discussions, fixtures, examples,
or pull requests.

This includes:

- secrets, credentials, tokens, keys, or `.env` files;
- customer confidential source material;
- raw PII;
- pricing, quotations, contracts, or financial calculation data;
- internal paths, names, handles, or identifiers that should not be public.

Use redacted examples or synthetic fixtures instead.

## Security Reports

For security-sensitive concerns, follow the [Security Policy](SECURITY.md).

Do not publish exploit details, private customer material, secrets, or raw
production data in a public issue.

## Support Boundaries

Axiom-PMO can help validate PMO artifacts, evidence structure, and governance
readiness. It does not replace human product approval, QA approval, security
review, legal review, release approval, or production deployment authority.
